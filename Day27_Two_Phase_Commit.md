# рҹӨқ Two-Phase Commit (2PC): The Distributed Transaction Protocol

*How Amazon Ensures Your Payment, Inventory, and Shipping Never Get Out of Sync*

---

## Introduction

You're buying an iPhone on Amazon. Three things must happen **atomically**:

1. **Payment Service**: Deduct вӮ№99,989 from your account
2. **Inventory Service**: Reserve 1 iPhone from stock
3. **Shipping Service**: Create shipping label and schedule pickup

**The Critical Rule:** Either ALL succeed, or NONE happen. No half-updated state!

Imagine if payment was deducted but inventory wasn't reserved. Or inventory was reserved but shipping wasn't created. Chaos.

This is the problem **Two-Phase Commit (2PC)** solves.

```yaml
Customer Action:
  User: John Doe
  Order: iPhone 15 (вӮ№79,999) + AirPods (вӮ№19,990)
  Total: вӮ№99,989
  Payment: Credit Card ending 1234

Resources That Must Be Updated ATOMICALLY:
  1. Payment Service  вҶ’ Deduct вӮ№99,989 from customer's account
  2. Inventory Service вҶ’ Reduce iPhone stock by 1, AirPods stock by 1
  3. Shipping Service  вҶ’ Create shipping label and schedule pickup

Critical Rule: Either ALL succeed, or NONE happen!
```

## The Big Problem: What If Coordinator Crashes?

### Scenario: Coordinator Dies After Some Commits

```
Timeline:
  1. Coordinator receives all YES votes
  2. Coordinator tells Payment: COMMIT вң“
  3. Coordinator tells Inventory: COMMIT вң“
  4. Coordinator CRASHES before telling Shipping: COMMIT

Now What?
  вң… Payment deducted (вӮ№99,989 gone!)
  вң… Inventory reduced (iPhone reserved!)
  вқҢ Shipping not created вҶ’ Order can't ship!

Customer paid, but no delivery scheduled!
```

### Solution: Write-Ahead Log + Recovery

```sql
-- Coordinator's Write-Ahead Log (WAL)
CREATE TABLE transaction_log (
    transaction_id VARCHAR(100),
    state ENUM('PREPARING', 'PREPARED', 'COMMITTING', 'COMMITTED', 'ABORTING', 'ABORTED'),
    participants JSON,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);

-- BEFORE sending any PREPARE:
INSERT INTO transaction_log 
VALUES ('ORD-12345', 'PREPARING', 
        '["payment","inventory","shipping"]', NOW(), NOW());

-- AFTER receiving all YES votes, BEFORE sending COMMIT:
UPDATE transaction_log 
SET state = 'COMMITTING', updated_at = NOW()
WHERE transaction_id = 'ORD-12345';

-- On RECOVERY after crash:
SELECT * FROM transaction_log 
WHERE state IN ('PREPARING', 'COMMITTING');

-- Recovery Logic:
-- If state = 'COMMITTING' вҶ’ Must complete COMMIT to all participants
-- If state = 'PREPARING' (and old) вҶ’ Safe to ABORT
```

## Performance Impact

```yaml

  в”ңв”Җв”Җ Each participant must persist state
```

## When to Use 2PC

```yaml
вң… USE 2PC When:
  - Banking/financial transactions (can't have half a transfer)
  - Flight/hotel bookings (atomic package deals)
  - Inventory management with strict consistency
  - Cross-database transactions
  - Compliance requirements (audit trail needed)

вқҢ AVOID 2PC When:
  - High throughput needed (use eventual consistency)
  - Services are geographically distributed (latency kills)
  - One participant is unreliable (cascading failures)
  - You can design around the need
  - Social media, analytics, logging (eventual consistency OK)

рҹ”„ CONSIDER ALTERNATIVES:
  - Saga Pattern (compensating transactions)
  - Eventual consistency with reconciliation jobs
  - Idempotent APIs + retry logic
  - Event-driven architecture
```
